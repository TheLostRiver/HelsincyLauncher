# Tauri Code Comment Standard

> Status: local draft v1
> Date: 2026-05-04
> Scope: repository-level code comment rules for Rust, TypeScript, TSX, JavaScript, and adjacent source files
> Related: `CONTRIBUTING.md`, `docs/README.md`, `docs/TauriArchitecturePrinciplesDesign.md`, `docs/ModuleDocumentationStandard.md`
> Focus: comment coverage, syntax split, selective body comments, concurrency-risk annotation, review rules

---

## 1. Purpose

这份文档只解决一件事：

把仓库里的“代码注释该写到什么程度、用什么形式写、哪些情况必须写、哪些情况不该写”收敛成一个独立且可执行的标准。

它要同时避免两个极端：

1. 把所有代码都机械注释一遍，制造低信号噪声。
2. 以“代码应当自解释”为理由，放任边界、不变量、风险点和并发约束完全不写。

本仓库不追求“注释越少越好”，也不追求“注释覆盖率越高越好”。

本仓库追求的是：

1. 明显代码不写废话注释。
2. 承载语义、边界、风险和约束的代码必须有高信号注释。

---

## 2. Non-goals

这份标准不负责：

1. 要求仓库当前所有旧代码立刻补齐注释。
2. 代替良好命名、良好类型设计或模块文档。
3. 强制每个私有局部变量都写注释。
4. 让 Rust、TypeScript 和 TSX 全部使用同一种注释语法。
5. 用长篇注释掩盖不该存在的复杂实现。

---

## 3. Core Principles

### 3.1 Comment The Why, Boundary, And Risk

注释首先解释这些内容：

1. 这个声明或逻辑负责什么。
2. 它的边界、不变量或前置条件是什么。
3. 为什么必须这样实现。
4. 如果改错了，风险会落在哪里。

如果一段注释只是把代码字面含义再说一遍，它通常不应该存在。

### 3.2 Obvious Code Stays Uncommented

一眼就能看懂的代码，不要求额外注释。

例如：

1. 明确命名的纯映射。
2. 无副作用的短小私有 helper。
3. 纯展示组件中的直白渲染分支。

### 3.3 Declaration Coverage Is Broader Than Body Coverage

声明级注释默认比函数体内注释覆盖更广。

也就是说：

1. 模块、类型、接口、函数、关键字段通常应有声明级注释。
2. 函数体内注释只在关键复杂逻辑处出现，不默认铺满函数体。

### 3.4 Concurrency Is A High-risk Exception

并发、多线程、共享状态、锁、取消、恢复、任务调度这类代码，不按“普通复杂代码”处理。

这类代码的注释要求更高，因为一旦理解错误，后果通常比普通业务分支更严重。

### 3.5 Comments Must Age With The Code

代码语义变了，相关注释必须在同一改动里一起更新。

过期注释不是文档资产，而是缺陷。

## 4. Comment Layers

| Surface | Default Rule | Can Omit When | Must Explain |
|--------|--------------|---------------|--------------|
| module / file entry | 默认要求在模块入口、边界文件、宿主入口、runtime/driver/adapter 入口提供注释 | 纯叶子文件且其职责已被所属模块文档完整覆盖 | 职责、拥有的状态、依赖方向、非目标、风险边界 |
| type / interface / trait / component | 对公开声明和语义重要的内部声明默认要求注释 | 纯本地、直白、短生命周期的数据壳 | 目的、生命周期、不变量、对外语义 |
| function / method / hook | 对公开函数、关键内部函数、带副作用或状态迁移的函数默认要求注释 | 私有纯 helper 且名字、签名、调用点都已自解释 | 职责、前置条件、副作用、失败语义、返回语义 |
| field / member | 对公开字段、关键状态字段、生命周期字段、缓存字段、并发字段默认要求注释 | 纯显然 DTO 字段，且类型名和字段名已完整表达语义 | 含义、单位、允许值、所有权、生命周期、并发语义 |
| function-body block | 默认不要求 | 逻辑直白且顺序、约束、风险都已明显 | 为什么要这样做、顺序为何重要、改错风险 |
| warning / concurrency note | 涉及锁、原子、共享状态、取消、恢复、幂等、防重、竞态风险时强制要求 | 不涉及并发或共享状态 | 资源保护目标、同步策略、顺序要求、失败后果 |

结论可以压缩成一句话：

默认要求覆盖到模块、类型、函数和关键字段层级；函数体内注释只服务于重要逻辑；并发注释按高风险区单独提升要求。

## 5. Syntax Rules By Language

### 5.1 TypeScript / TSX / JavaScript

这类文件优先使用 Doxygen 风格或兼容 TSDoc/JSDoc 的块注释来承载声明级注释。

推荐用于：

1. 导出的函数、组件、hooks、类型、接口。
2. 模块入口说明。
3. 需要写清参数、返回值、副作用的声明。

规则：

1. 需要结构化信息时，使用 `@param`、`@returns`、`@throws`、`@remarks` 等标签。
2. 局部实现说明仍然使用普通单行注释，不要为了统一外观把所有局部注释都改成块注释。
3. 不要给每个局部变量都写 JSDoc 风格注释。

### 5.2 Rust

Rust 不采用“所有声明一律 Rustdoc”的做法。

规则如下：

1. `//!` 主要用于 crate 或 module 入口的职责、边界和注意事项。
2. `///` 主要用于 public API，或少数确实需要结构化文档的关键边界声明。
3. 私有实现、关键内部状态、函数体内重点说明，优先使用普通注释，而不是把所有私有项都改成 Rustdoc。

这条规则的目标不是减少 Rust 注释，而是避免语法和语义错位。

### 5.3 Shared Rule

语法可以分语言，语义标准不分语言。

无论使用哪种注释语法，都必须优先解释：

1. 职责
2. 边界
3. 不变量
4. 风险
5. 并发或恢复语义

## 6. Required Declaration Content

### 6.1 Module Or File Entry

下面这些文件通常应有文件级或模块级说明：

1. 宿主入口
2. command 聚合面
3. composition root
4. runtime / driver / adapter 入口
5. 承担模块边界的 facade 或 contract 文件

应至少说明：

1. 负责什么。
2. 不负责什么。
3. 依赖方向是什么。
4. 若有状态或并发行为，风险边界在哪里。

### 6.2 Type / Interface / Trait / Component

应优先说明：

1. 这个类型或组件的核心语义。
2. 生命周期或使用边界。
3. 是否承载不变量、状态机、缓存、恢复或并发语义。

### 6.3 Function / Method / Hook

应优先说明：

1. 这个函数解决什么问题。
2. 关键前置条件是什么。
3. 是否有副作用、IO、状态迁移、持久化、恢复或失败投影。
4. 若返回值可能被误解，返回语义是什么。

### 6.4 Field / Member

字段注释不是为了“把字段名翻译一遍”，而是为了补充名字和类型没有说清的语义。

尤其是下面这些字段，默认应有注释：

1. 持久化或 runtime 状态字段。
2. 缓存、失效、预热、恢复、重试相关字段。
3. 表示所有权、生命周期、句柄、租约、版本、优先级、单位的字段。
4. 锁、原子变量、通道、共享句柄等并发字段。

---

## 7. Function-body Comment Rules

函数体内注释默认不是“到处都写”，而是只在下面这些情况出现：

1. 顺序要求重要，改动顺序会破坏语义。
2. 存在不明显的前置条件或不变量。
3. 涉及恢复、回滚、重试、补偿、缓存失效或状态迁移。
4. 存在安全、性能、兼容性或历史坑位的特殊写法。
5. 不写就很容易误改出 bug。

函数体内注释应该解释：

1. 为什么这段逻辑需要这样安排。
2. 哪个风险点被这段逻辑压住了。
3. 哪个条件改变后，这段逻辑可能需要一起调整。

函数体内注释不应该做这些事：

1. 逐行翻译代码。
2. 重复 if/else 的字面意思。
3. 用长段说明掩盖糟糕命名。

---

## 8. Concurrency And Multi-threading Rules

### 8.1 When The Rule Applies

只要代码涉及下列任一情况，就进入并发高风险区：

1. 多线程或后台任务并发执行。
2. `Mutex`、`RwLock`、原子变量、通道、共享句柄。
3. 取消、超时、重试、防重、幂等。
4. job runtime、恢复、租约、去重、状态机并发推进。
5. 多个任务对同一资源的读写顺序依赖。

### 8.2 Required Content

这类代码的注释至少要解释其中的大部分关键点：

1. 这段并发逻辑保护什么资源或不变量。
2. 谁拥有状态，谁只读，谁可写。
3. 为什么选择当前同步原语，而不是别的同步方式。
4. 是否存在锁顺序要求。
5. 是否允许重入或重复触发。
6. 取消、超时、失败或恢复时状态如何收敛。
7. 为什么当前实现不会明显地产生竞态、死锁或重复执行。

### 8.3 Field-level Requirement

对于承载并发语义的字段，不能只在使用处解释。

下面这些字段通常应在声明处写注释：

1. 锁保护的共享状态。
2. 记录 worker 生命周期的句柄。
3. 原子标志位、版本号、取消标记。
4. 表示所有权或租约的共享 token。

### 8.4 Order-sensitive Flow Requirement

如果正确性依赖跨线程或跨任务的操作顺序，那么顺序约束必须在编排点附近写明。

不要把这类关键说明只埋在深层 helper 里。

---

## 9. Forbidden Patterns

禁止下面这些低质量注释：

1. 把标识符再说一遍的废话注释。
2. 已经和当前实现不一致的历史注释。
3. 把明显代码硬写成逐行讲解。
4. 用注释替代应该拆分的函数或应该改善的命名。
5. 在高风险并发代码里只写“加锁保护”这种没有信息量的话。

---

## 10. Review Checklist

评审注释时，至少检查下面这些问题：

1. 关键声明是否解释了职责和边界。
2. 高风险逻辑是否解释了不变量和风险点。
3. 并发代码是否把同步策略和顺序要求讲清楚。
4. 是否存在明显可以删掉的废话注释。
5. 本次改动是否已经让旧注释失效。
6. 注释语法是否符合当前语言和可见性层级。

---

## 11. Rollout Rule

这份标准的执行方式不是“先停下所有开发，统一补注释”，而是：

1. 新增代码从现在开始遵守。
2. 修改旧代码时，顺手修正同一语义面的注释缺口。
3. 高风险并发、恢复、状态机相关代码优先补齐。
4. 明显代码不为凑覆盖率而补注释。

一句话总结：

声明级注释应覆盖语义边界；函数体内注释只服务于关键复杂逻辑；并发相关逻辑按高风险区单独提高要求；Rustdoc 主要用于模块入口和 public API，而不是 Rust 全域统一语法。
